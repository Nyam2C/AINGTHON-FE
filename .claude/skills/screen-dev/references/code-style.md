# 코드 스타일 & review 기준

CLAUDE.md의 4대 원칙(코딩 전 사고 / 단순함 / 외과적 변경 / 목표 기반 실행)을 review에서 검증한다.

## review 체크리스트

### 1. 단순함

- [ ] 한 번만 쓰는 추상화가 있는가? → 인라인으로 풀기
- [ ] 50줄로 충분한데 200줄로 작성됐는가? → 단순화
- [ ] 요청되지 않은 "유연성" 옵션(`config`, `options` prop 등)이 있는가? → 제거
- [ ] 일어날 수 없는 시나리오에 대한 방어 코드가 있는가? → 제거

### 2. 외과적 변경

- [ ] 이번 화면과 무관한 파일이 수정됐는가? → 사용자에게 보고
- [ ] 인접 코드를 "개선"한 흔적이 있는가? → 되돌리기
- [ ] 사용하지 않는 import / 변수 / 타입이 남아있는가? → 제거

### 3. 폴더/명명 규칙 준수 (`architecture.md` 기준)

- [ ] 컴포넌트가 올바른 디렉토리에 있는가 (screens/ vs components/[feature]/ vs components/common/)
- [ ] 파일명이 컨벤션을 따르는가
- [ ] 컴포넌트가 axios를 직접 import 하지 않는가
- [ ] 서버 상태가 useEffect로 fetch되지 않고 TanStack Query를 쓰는가

### 4. 타입 안전성

- [ ] `any` 사용 여부 — 있으면 `unknown` + narrowing으로 교체
- [ ] API 응답 타입이 `types/`에 정의되어 있는가
- [ ] 컴포넌트 props에 타입이 명시되어 있는가

### 5. 테스트

- [ ] 작성된 테스트가 통과하는가 (`pnpm test`)
- [ ] 핵심 사용자 시나리오를 커버하는가
- [ ] 구현 세부사항(internal state, className)을 테스트하고 있지 않는가

### 6. Tailwind

- [ ] 인라인 `style` prop이 있는가? → Tailwind 클래스로 교체
- [ ] 같은 클래스 조합이 3회 이상 반복되는가? → 컴포넌트 추출 고려

## review 출력 형식

review 에이전트는 다음 형식으로 보고한다:

```markdown
## Review 결과

### 발견 사항
- [심각도] 위치 — 문제 — 적용한 수정 (또는 사용자 결정 요청)

### 적용한 변경
- 파일:라인 — 변경 내용

### 보류한 변경 (사용자 결정 필요)
- 항목 — 이유

### 테스트 결과
- 통과: N개
- 실패: N개 (있으면 원인)
```

## 리팩토링 시 지킬 것

- **기존 동작을 유지한다.** 리팩토링 전후로 모든 테스트가 동일하게 통과해야 한다.
- **한 번에 한 종류만.** 이름 바꾸기, 함수 추출, 인라인화를 동시에 하지 않는다.
- **테스트가 없으면 리팩토링하지 않는다.** 안전망 없는 리팩토링은 회귀를 만든다.

## 절대 하지 말 것

- 사용자가 요청하지 않은 "보너스 기능" 추가
- 기존 코드의 무관한 부분 정리
- 가상의 미래 요구사항을 위한 추상화 도입
- 동작하는 코드의 스타일만 바꾸는 변경
